# 🤖 公司午餐訂購系統：AI 菜單辨識功能 深度架構與演進報告

本報告針對系統中最新開發的「AI 菜單辨識功能」進行徹底的技術盤點。此功能不僅串接了目前 Google 最先進的多模態大型語言模型，更在過程中披荊斬棘，解決了各種極端情境與雲端運算限制，最終演化成一個極具韌性的工業級實作。

---

## 🏗️ 1. 系統架構與業務流程 (Architecture & Workflow)

整個 AI 菜單辨識的資料流經過了嚴密的層層把關與優化：

1. **前端使用者介面 (`AIMenuScanner.js`)**
   - 支援拖曳或點選上傳圖片。
   - **本機預處理**：在將圖片傳送至伺服器前，前端會自動透過 HTML5 Canvas 將過大的照片實時壓縮與縮放（確保大小 < 4MB）。此舉完美避開了 Firebase Callable HTTP Payload 過大的錯誤，同時加快上傳速度。
2. **安全通訊層 (`menuAIService.js`)**
   - 採用 Firebase Callable Functions 機制呼叫後端，自帶使用者身分驗證 (`request.auth`)，無需手動傳遞 Token。
3. **後端辨識核心 (`functions/menuAI.js`)**
   - 在亞洲節點 (`asia-east1`) 執行，與資料庫同源，減少延遲。
   - 導入 Google 最新官方推薦的 `@google/genai` 套件，透過 `gemini-2.5-flash` 模型提供性價比極高且極速的視覺理解分析。
   - 透過精心設計的 Prompt Engineering，嚴格要求 AI 不得擅自進行英文翻譯，並且將尺寸（如大杯/中杯）精確 mapping 至 `variants` 結構中。
4. **前端差異整合 (`MenuEditor.js`)**
   - 收到 AI 回傳的 JSON 後，系統會自動將它與目前店家已存在的菜單進行比對。
   - 於畫面上清楚標示「新增」了哪些項目，並讓管理員在確認無誤後，一次性寫入資料庫。

## 🛡️ 2. 安全防禦與頻率限制 (Security & Rate Limiting)

AI 資源極其寶貴，我們為此系統實作了多重保護：
- **存取權限控管**：結合前端的 UI 遮罩與後端的 JWT 驗證，未登入或非 Admin (超級管理員/管理員) 角色的人絕對無法觸發辨識，也無法通過 `firestore.rules` 改寫菜單。
- **記憶體級別防刷機制**：Cloud Function 中內建基於 `UID + IP` 的 Rate Limiter（60秒內最多呼叫 15 次）。這有效防止了惡意腳本短時間內爆刷 API，保護我們 GCP 專案的免費額度與計費盲區。

## ⚡ 3. 歷經的極端挑戰與黑科技解法 (V1~V9 演化史)

在實際導入過程中，我們遭遇了各種 LLM 與 Firebase 系統底層的物理限制。全數已被我們特製的方案一一擊破：

### ❌ 挑戰一：跨域 (CORS) 與 Firebase 隱藏節點坑
初次上線時遇到了嚴重的 CORS 錯誤。
👉 **解法**：這是因為 Cloud Functions V2 在定義 Region 時的順序陷阱。我們將 `region: "asia-east1"` 嚴格定義在 `onCall` 內部參數中，確保前後端通訊區域完全一致。同時棄用舊環境變數，改用最新的 `GCP_PROJECT` 確保 Vertex AI 認證成功。

### ❌ 挑戰二：舊版 SDK 的多位元組中文字亂碼
隨著菜單增長，舊版 `@google-cloud/vertexai` SDK 在接收長度極長的字串時，會從 byte 層面不當切割，導致後段跑出 `???` 等中文字亂碼。
👉 **解法**：忍痛全面昇級替換至 Google 最新釋出的 `@google/genai` 原生 SDK，從底層修復了 UTF-8 串流截斷的編碼問題。

### ❌ 挑戰三：AI 思考模式 (`<think>`) 干擾與 Markdown 外殼
Gemini 此類新一代思考模型有時會把它的推論過程寫進回傳中，導致純粹的 `JSON.parse` 當場崩潰。
👉 **解法**：導入自訂的防禦性解析法。程式碼在解析前，會使用 `lastIndexOf('</think>')` 強制移除所有先皇敘述，並透過 Regex 剝除 ```json 的 Markdown 外殼，確保留下來的只有乾淨程式碼。

### ❌ 挑戰四：無可避的 Token 絕對上限與「斷尾殘缺 JSON」
百項級別的大菜單，AI 產出到一半必會觸發 `maxOutputTokens: 8192` 的物理硬上限，導致 JSON 在一半被「喀嚓」截斷，連最後一個右括號都沒有。
👉 **解法**：導入了 `jsonrepair` 特效藥。當發現 LLM 產生出缺少結尾括號與引號的殘缺文字時，`jsonrepair` 會在一瞬間自動反向回溯並強制閉合所有結構！這個機制成功救出了 80~90% 已經辨識完成的陣列，再也沒有「格式異常」的遺憾。

### ❌ 挑戰五：Firestore 歷史紀錄毒化與文件索引爆炸
在百項商品辨識成功後，點擊儲存卻被 Firestore 彈出 `Missing or insufficient permissions` 錯誤。這不是因為權限被拔。
👉 **解法**：這是因為舊版的 `updateStore` 每次都會將一整坨「上千個欄位」的歷史操作紀錄再次寫回文件，導致文件突破了 1MB / 20,000 索引的地獄上限。我們開發了歷史清創機制，會在下一次存檔前，強制將舊的臃腫歷史陣列壓縮成 `[Cleaned legacy info]`，從此徹底根絕文件大小爆炸的問題。
